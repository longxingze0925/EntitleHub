# 用户管理后台系统开发入口

本文是整个项目文档目录的入口文件。

目标不是把所有细节重复一遍，而是告诉 AI 或开发者：

- 这个系统到底要做什么；
- 开发前必须先读哪些文档；
- 每次开发任务应该怎么拆；
- 哪些安全规则不能省；
- 做完后怎么验收。

本目录下的文档服务于一个目标：从零开发一套自己的用户管理后台系统，不是改 FerrisKey，也不是把 FerrisKey 和现有系统合并。

---

## 1. 系统定位

这是一个完整的自有用户管理、授权、客户端通信安全平台。

它要解决的核心问题：

1. 管理后台能安全地管理用户、角色、权限、租户、客户端、设备、授权、版本、脚本和审计日志。
2. 客户端能安全登录、激活、续期、上报状态、获取配置、下载更新、获取脚本。
3. 服务端能识别客户端身份，防止伪造请求、重放请求、盗用令牌、越权访问。
4. 文件、热更新包、脚本下发必须有签名校验，客户端不能盲目信任下载内容。
5. 整个系统要能长期维护，不能只做一个能跑的 demo。

一句话：后台管人、管权限、管客户端、管授权；客户端安全通信；所有关键操作可审计、可撤销、可追踪。

## 2. 推荐技术栈

后端：

- Rust
- Axum
- PostgreSQL
- sqlx
- Redis
- object storage，例如 MinIO 或 S3

后台前端：

- React
- TypeScript
- Ant Design 或同级后台 UI 框架

部署：

- Docker Compose 起步
- 后续可升级 Kubernetes
- Nginx 或 Caddy 做 HTTPS 入口

安全基础：

- HTTPS
- HttpOnly Secure SameSite Cookie
- CSRF 防护
- RBAC 权限控制
- MFA 管理员二次验证
- 短 Access Token
- Refresh Token 轮换与撤销
- Ed25519 请求签名
- JWKS 公钥发布与验签
- 审计日志

## 4. AI 开发总规则

AI 每次开发任务必须遵守以下规则。

### 4.1 先读后改

不能直接写代码。

每次任务开始前，先确认：

- 任务属于哪个模块；
- 涉及哪些表；
- 涉及哪些接口；
- 是否影响权限；
- 是否影响客户端兼容；
- 是否影响安全策略；
- 是否需要数据库迁移。

### 4.2 小步开发

一次任务只做一个明确目标。

正确例子：

- 实现管理员登录接口；
- 实现 Refresh Token 轮换；
- 实现设备激活；
- 实现角色权限检查中间件。

错误例子：

- 一次性实现整个后台；
- 一次性写完用户、权限、授权、客户端、更新系统；
- 边写功能边大规模重构。

### 4.3 表结构变更必须走迁移

不能手工改数据库。

任何表结构变化都必须：

1. 新增 migration 文件；
2. 写清楚字段、索引、约束；
3. 支持重复部署；
4. 本地执行迁移；
5. 更新 `数据库表结构.md`。

### 4.4 接口变更必须更新文档

任何 API 改动都必须同步更新 `API接口文档.md`。

包括：

- URL；
- method；
- 请求字段；
- 响应字段；
- 错误码；
- 权限要求；
- 是否需要签名；
- 是否需要 CSRF；
- 是否允许客户端访问。

### 4.5 安全逻辑必须有测试

以下逻辑不能只靠手测：

- 密码校验；
- session 创建和失效；
- refresh token 轮换；
- refresh token 复用检测；
- RBAC 权限判断；
- 租户隔离；
- 请求签名验证；
- nonce 防重放；
- 文件签名验证；
- 管理员 MFA；
- 审计日志落库。

## 5. 模块边界

建议按以下模块拆分。

### 5.1 auth

负责：

- 管理员登录；
- 管理员登出；
- session；
- CSRF；
- MFA；
- 密码哈希；
- 登录失败限制。

不能负责：

- 业务用户资料；
- 客户端授权；
- 软件更新。

### 5.2 iam

负责：

- 管理员账号；
- 角色；
- 权限；
- 组织或租户；
- RBAC 检查。

不能负责：

- 客户端设备身份；
- refresh token 轮换。

### 5.3 user

负责：

- 业务用户；
- 用户状态；
- 用户绑定关系；
- 用户登录历史；
- 用户封禁。

### 5.4 client

负责：

- 客户端应用；
- 客户端版本；
- 设备；
- 设备激活；
- 客户端会话；
- 客户端请求签名。

### 5.5 license

负责：

- 授权计划；
- 授权码；
- 授权绑定；
- 授权过期；
- 授权撤销。

### 5.6 update

负责：

- 版本管理；
- 更新包上传；
- 更新包 hash；
- 更新包签名；
- 灰度发布；
- 防降级。

### 5.7 script

负责：

- 脚本管理；
- 脚本版本；
- 脚本签名；
- 脚本下发策略；
- 脚本撤销。

### 5.8 audit

负责：

- 管理后台操作日志；
- 客户端关键行为日志；
- 安全事件；
- 审计查询。

### 5.9 platform

负责：

- 配置；
- 健康检查；
- 系统状态；
- 定时任务；
- 密钥轮换任务。

## 6. 开发阶段

### M0：项目骨架

目标：

- 后端项目能启动；
- 数据库能连接；
- Redis 能连接；
- 健康检查接口可用；
- 基础配置加载完成；
- 日志格式统一。

验收：

- `GET /healthz` 返回正常；
- `GET /readyz` 能检查数据库和 Redis；
- 本地测试能跑通。

### M1：管理员认证

目标：

- 管理员登录；
- HttpOnly Cookie session；
- CSRF；
- 登出；
- 密码哈希；
- 登录失败限制。

验收：

- Cookie 不能被 JavaScript 读取；
- 非法 CSRF 请求被拒绝；
- 错误密码不会泄露用户是否存在；
- 登录失败过多会被限制。

### M2：权限系统

目标：

- 管理员；
- 角色；
- 权限；
- RBAC；
- 租户隔离。

验收：

- 没权限访问返回 403；
- A 租户不能读写 B 租户数据；
- 权限检查不能只放在前端。

### M3：业务用户管理

目标：

- 用户增删改查；
- 用户状态；
- 用户封禁；
- 用户查询；
- 用户审计。

验收：

- 所有用户数据带 `tenant_id`；
- 删除优先软删除；
- 敏感操作写审计日志。

### M4：客户端身份和设备激活

目标：

- 客户端应用注册；
- 设备注册；
- 设备密钥；
- 设备激活；
- 客户端会话。

验收：

- 设备私钥不能上传服务端；
- 服务端只保存设备公钥；
- 未激活设备不能访问受保护接口。

### M5：客户端令牌和请求签名

目标：

- Access Token；
- Refresh Token；
- Refresh Token 轮换；
- Refresh Token 复用检测；
- 请求签名；
- nonce 防重放。

验收：

- Access Token 有短有效期；
- Refresh Token 只保存 hash；
- 旧 Refresh Token 再次使用会触发会话撤销；
- 同一个 nonce 重放会被拒绝。

### M6：授权系统

目标：

- 授权计划；
- 授权码；
- 授权绑定；
- 授权过期；
- 授权撤销。

验收：

- 授权状态由服务端判断；
- 客户端不能自己决定是否有效；
- 撤销后客户端续期失败。

### M7：更新和脚本下发

目标：

- 更新包上传；
- hash 校验；
- 服务端签名；
- 客户端验签；
- 脚本版本；
- 脚本签名；
- 灰度策略。

验收：

- 客户端下载后必须验证 hash 和签名；
- 旧版本不能覆盖新版本；
- 被撤销脚本不能继续下发。

### M8：后台前端

目标：

- 登录页；
- 管理员管理；
- 角色权限；
- 用户管理；
- 设备管理；
- 授权管理；
- 更新管理；
- 脚本管理；
- 审计日志。

验收：

- 前端只做展示和交互；
- 权限最终由后端判断；
- 敏感操作需要二次确认；
- 表格查询、分页、筛选稳定。

### M9：安全测试和上线

目标：

- 单元测试；
- 集成测试；
- 安全测试；
- 部署脚本；
- 备份策略；
- 回滚流程；
- 监控告警。

验收：

- `安全测试清单.md` 中 MVP 必测项通过；
- 备份能恢复；
- 发布失败能回滚；
- 关键安全事件有告警。

### 7.2 开发中

- 不顺手重构无关模块；

- 不硬编码密钥；

必须输出：

```text

验证：

下一步：
```

### 7.4 验证失败时

不能只说失败。

必须说明：

- 失败命令；
- 失败原因；
- 已经排除什么；
- 建议下一步怎么处理。

## 8. 安全红线

以下事情不能做。

### 8.1 管理后台

不能：

- 把管理员 token 放 localStorage；
- 只靠前端隐藏按钮控制权限；
- 关闭 CSRF；
- session 永不过期；
- 密码明文保存；
- 登录错误直接提示“用户不存在”；
- 管理员敏感操作不写审计日志。

必须：

- 使用 HttpOnly Secure SameSite Cookie；
- 服务端检查权限；
- 密码使用 Argon2id 或同级算法；
- 登录失败限流；
- MFA 支持；
- 审计日志不可随便删除。

### 8.2 客户端通信

不能：

- 长期 Access Token；
- Refresh Token 明文入库；
- 不校验请求时间戳；
- 不校验 nonce；
- 客户端密钥写死在程序里；
- 热更新包下载后直接执行；
- 脚本下发后直接运行。

必须：

- Access Token 短有效期；
- Refresh Token hash 存储；
- Refresh Token 轮换；
- 设备私钥只在客户端本地；
- 请求签名包含 method、path、timestamp、nonce、body hash；
- 更新包必须验证 hash 和签名；
- 脚本必须验证签名和版本。

### 8.3 数据库

不能：

- 多租户表缺少 `tenant_id`；
- 查询不带租户条件；
- 删除关键数据不留审计；
- 用字符串拼 SQL；
- 忽略唯一约束。

必须：

- 所有租户数据强制隔离；
- 使用 sqlx 参数化查询；
- 关键字段加索引；
- 敏感数据加密或 hash；
- migration 可重复部署。

## 11. 日志和审计

普通日志用于排查问题。

审计日志用于追踪责任。

两者不能混为一谈。

普通日志可以记录：

- request id；
- trace id；
- 接口路径；
- 状态码；
- 耗时；
- 错误类型。

普通日志不能记录：

- 密码；
- access token；
- refresh token；
- session id；
- 设备私钥；
- 签名私钥；
- 完整授权码；
- 完整身份证、手机号等敏感信息。

审计日志必须记录：

- 操作人；
- 操作对象；
- 操作类型；
- 操作时间；
- 来源 IP；
- user agent；
- 操作结果；
- 失败原因；
- request id。

## 12. 验收标准

一个阶段是否完成，不看“代码写了多少”，只看是否满足验收。

### 后端接口验收

必须满足：

- 有权限检查；
- 有输入校验；
- 有错误码；
- 有日志；
- 敏感操作有审计；
- 有单元测试或集成测试；
- API 文档已更新。

### 数据库验收

必须满足：

- migration 可执行；
- 约束正确；
- 索引合理；
- 外键或逻辑关联明确；
- 多租户数据隔离；
- 软删除策略明确。

### 前端验收

必须满足：

- 页面能正常打开；
- 登录态失效能回登录页；
- 无权限页面不能访问；
- 表格分页、搜索、筛选可用；
- 操作失败有明确提示；
- 敏感操作有确认。

### 客户端 SDK 验收

必须满足：

- 激活流程可用；
- token 自动刷新；
- refresh token 轮换；
- 请求签名正确；
- 离线缓存策略明确；
- 更新包验签；
- 脚本验签。

MVP 必须包含：

- 管理员登录；
- 管理员 session；
- CSRF；
- RBAC；
- 租户隔离；
- 用户管理；
- 客户端应用管理；
- 设备激活；
- Access Token；
- Refresh Token 轮换；
- 请求签名；
- 授权绑定；
- 更新包签名验证；
- 审计日志；
- 基础部署；
- 备份和回滚。

MVP 可以暂缓：

- 复杂组织架构；
- SSO；
- WebAuthn；
- 多区域部署；

## 15. 常见错误

### 错误 1：先写页面，后补权限

这是错的。

权限必须先在后端模型和接口层设计好。

前端只是展示，不是安全边界。

### 错误 2：客户端只用 HTTPS 就认为安全

HTTPS 只能保护传输过程。

它不能解决：

- token 被盗；
- 请求被重放；
- 客户端被伪造；
- 更新包被替换；
- 脚本被篡改。

所以客户端还需要：

- 短 token；
- refresh token 轮换；
- 设备密钥；
- 请求签名；
- nonce；
- 文件签名验证。

### 错误 3：Refresh Token 不轮换

长期有效的 refresh token 一旦泄露，就很难发现。

正确做法：

- 每次刷新都签发新的 refresh token；
- 旧 refresh token 立即失效；
- 旧 token 再次出现，认为可能被盗；
- 立刻撤销对应会话。

### 错误 4：脚本下发不签名

脚本属于高危内容。

客户端执行脚本前必须确认：

- 脚本来自可信服务端；
- 内容没有被篡改；
- 版本没有被回滚；
- 下发策略允许当前设备执行。

### 错误 5：多租户只在前端筛选

这是严重漏洞。

租户隔离必须在后端查询层和权限层强制执行。

## 17. 最终判断标准

这个系统做得好不好，不是看功能列表多不多。

真正要看：

- 管理员账号被攻击时有没有保护；
- 客户端请求能不能被伪造；
- token 泄露后能不能发现和撤销；
- 设备被盗后能不能禁用；
- 更新包被替换后客户端会不会执行；
- 脚本被篡改后客户端会不会运行；
- 权限越权能不能被挡住；
- 租户数据会不会串；
- 出问题后能不能从审计日志追到原因；
- 发布失败后能不能回滚；
- 数据丢失后能不能恢复。

只要这些问题有明确答案，这个系统才算真正能用。
