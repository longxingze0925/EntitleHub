# 安全测试清单

本文档用于约束后台系统开发过程中的安全测试。每个安全功能上线前，必须完成对应测试。AI 开发时不能只实现功能接口，还必须补充安全测试或至少给出可手动验证步骤。

## 1. 测试原则

所有测试必须覆盖：

- 正常路径。
- 未认证路径。
- 无权限路径。
- 跨租户路径。
- 过期状态。
- 撤销状态。
- 重放攻击。
- 篡改请求。
- 限流触发。
- 审计记录。

重要原则：

```text
没有安全测试的安全功能，视为没有完成。
```

## 2. 认证安全测试

### 2.1 后台登录成功

步骤：

1. 创建 active 租户。
2. 创建 active team_member。
3. 使用正确邮箱密码登录。

预期：

- 返回用户信息。
- 设置 HttpOnly Cookie。
- 不返回可被前端保存的 access token。
- 写入 login_attempt 成功记录。

### 2.2 后台登录失败

场景：

- 邮箱不存在。
- 密码错误。
- 用户 disabled。
- 租户 suspended。

预期：

- 返回统一错误，不泄露账号是否存在。
- 密码错误记录失败次数。
- disabled 用户不能登录。
- suspended 租户用户不能登录。

### 2.3 登录限流

步骤：

1. 同一 IP 连续错误登录超过阈值。
2. 同一 email 连续错误登录超过阈值。

预期：

- 返回 429。
- 错误信息不泄露敏感信息。
- 限流 key 进入 Redis。

### 2.4 Cookie 属性

检查：

- `HttpOnly`
- `Secure`
- `SameSite`
- `Path`
- 合理过期时间

预期：

- 生产环境必须有 `Secure`。
- Cookie 不能被 JavaScript 读取。

## 3. Session 安全测试

### 3.1 未登录访问

接口：

```text
GET /api/auth/me
GET /api/admin/apps
```

预期：

- 返回 401。

### 3.2 撤销 session

步骤：

1. 登录获取 session。
2. 调用 logout。
3. 使用旧 Cookie 访问受保护接口。

预期：

- 返回 401。

### 3.3 改密码踢下线

步骤：

1. 同一用户登录两个浏览器。
2. 浏览器 A 修改密码。
3. 浏览器 B 继续访问后台。

预期：

- 浏览器 B session 被撤销或要求重新登录。

### 3.4 refresh token 轮换

步骤：

1. 使用 refresh token 刷新。
2. 再次使用旧 refresh token。

预期：

- 第一次成功。
- 第二次失败。
- 整个 session 被撤销。
- 记录安全审计。

## 4. CSRF 测试

### 4.1 无 CSRF token 修改资源

接口：

```text
POST /api/admin/apps
PUT /api/tenant
DELETE /api/admin/devices/{id}
```

预期：

- 返回 403。

### 4.2 错误 CSRF token

预期：

- 返回 403。
- 不执行业务变更。

### 4.3 GET 不要求 CSRF

预期：

- GET、HEAD、OPTIONS 不要求 CSRF。

## 5. CORS 测试

### 5.1 允许来源

请求 Origin：

```text
https://admin.example.com
```

预期：

- 返回允许 Origin。
- credentials 设置正确。

### 5.2 非法来源

请求 Origin：

```text
https://evil.example.com
```

预期：

- 不返回 Access-Control-Allow-Origin。
- OPTIONS 返回拒绝或不允许。

### 5.3 禁止通配符和 credentials 同时存在

预期：

```text
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

不能同时出现。

## 6. 权限测试

### 6.1 低权限访问高权限接口

viewer 访问：

```text
POST /api/admin/apps
POST /api/admin/licenses
POST /api/admin/devices/{id}/blacklist
```

预期：

- 返回 403。

### 6.2 developer 访问审计

预期：

- 返回 403。

### 6.3 owner 权限

预期：

- owner 可访问全部租户内资源。

### 6.4 禁用用户权限

步骤：

1. 用户登录。
2. 管理员禁用该用户。
3. 用户继续访问。

预期：

- 返回 403 或 401。
- 相关 session 被撤销。

## 7. 租户隔离测试

### 7.1 跨租户读取

步骤：

1. 租户 A 创建 app。
2. 租户 B 用户访问该 app id。

预期：

- 返回 404 或 403。
- 不能返回租户 A 的任何信息。

### 7.2 跨租户修改

租户 B 尝试更新租户 A 的：

- app
- license
- customer
- device
- release

预期：

- 全部失败。
- 审计记录异常访问。

### 7.3 列表接口隔离

预期：

- 租户 A 只能看到租户 A 数据。
- 租户 B 只能看到租户 B 数据。

## 8. 密码安全测试

### 8.1 密码哈希

检查数据库：

- 不能出现明文密码。
- hash 必须是 Argon2id 格式。

### 8.2 弱密码拒绝

测试：

- `123456`
- `password`
- `abcdefg`
- `Admin123`

预期：

- 全部拒绝。

### 8.3 重置密码

预期：

- 重置 token 只能使用一次。
- 过期 token 失败。
- 重置成功后旧 session 失效。

### 8.4 一次性 token

覆盖：

- 成员邀请 token。
- 管理员密码重置 token。
- 客户密码重置 token。
- 邮箱验证 token。

预期：

- 数据库只保存 token hash。
- token 只能使用一次。
- 过期 token 失败。
- 已撤销 token 失败。
- 使用成功后写入 `consumed_at`。
- 重复使用返回稳定错误码。

## 9. MFA 测试

### 9.1 启用 MFA

预期：

- secret 只返回一次。
- recovery code 只返回一次。
- 数据库存加密 secret 和 recovery hash。

### 9.2 MFA 登录

场景：

- 正确 TOTP。
- 错误 TOTP。
- 过期 TOTP。
- recovery code。

预期：

- 只有正确 TOTP 或未使用 recovery code 成功。

### 9.3 禁用 MFA

预期：

- 必须输入密码。
- 必须输入当前 TOTP 或 recovery code。

## 10. 客户端激活测试

### 10.1 正常激活

请求：

- 有效 app_key。
- 有效 license_key。
- 未绑定 machine_id。
- 有 device_public_key。

预期：

- 创建 device。
- 创建 client_session。
- 返回 access_token 和 refresh_token。

### 10.2 授权码错误

预期：

- 返回 401 或 400。
- 不创建 device。
- 不创建 session。

### 10.3 授权过期

预期：

- 激活失败。

### 10.4 设备数量超限

预期：

- 激活失败。
- 不创建新设备。

### 10.5 黑名单设备激活

预期：

- 激活失败。
- 记录审计或安全事件。

## 11. 客户端 session 测试

### 11.1 access token 过期

预期：

- 过期 token 访问失败。
- refresh 后可继续访问。

### 11.2 refresh token 轮换

预期：

- 刷新成功返回新 refresh token。
- 旧 refresh token 不能再次使用。

### 11.3 refresh token 复用检测

步骤：

1. 使用 refresh token A 刷新得到 B。
2. 再次使用 A。

预期：

- 复用失败。
- 撤销整个 client_session。
- B 也不能再刷新。

### 11.4 session 撤销

场景：

- 客户端 logout。
- 管理员解绑设备。
- 管理员拉黑设备。
- 授权被吊销。

预期：

- 相关 access token 不能继续访问。

## 12. 请求签名测试

### 12.1 正确签名

预期：

- 请求通过。

### 12.2 篡改 body

预期：

- body hash 不匹配。
- 返回 401 或 403。

### 12.3 篡改 path

预期：

- 验签失败。

### 12.4 过期 timestamp

预期：

- 请求失败。

### 12.5 重复 nonce

预期：

- 第一次成功。
- 第二次失败。

### 12.6 错误 device key

预期：

- 验签失败。

### 12.7 JWKS 只暴露公钥

步骤：

1. 请求 `GET /.well-known/jwks.json`。
2. 请求 `GET /api/client/apps/{app_key}/jwks`。

预期：

- 返回 key 都有 `kid`。
- 返回 key 不包含私钥。
- 不返回 `revoked` key。

### 12.8 签名 kid 不存在

步骤：

1. 构造带不存在 `kid` 的更新包或脚本签名。
2. 客户端 SDK 执行验签。

预期：

- SDK 刷新 JWKS。
- 刷新后仍找不到 `kid` 时拒绝安装或拒绝使用脚本。

### 12.9 密钥轮换兼容

步骤：

1. 使用旧 key 签发一个短期 token 或更新包。
2. 轮换出新 key。
3. 旧 key 标记为 `retiring`。
4. 同时验证旧签名和新签名。

预期：

- 旧签名在有效期内仍可验证。
- 新签名使用新 `active` key。
- 旧 key 过期后改为 `retired`。
- `revoked` key 不能再用于新签名。

## 13. 设备安全测试

### 13.1 拉黑设备

步骤：

1. 客户端已激活。
2. 管理员拉黑设备。
3. 客户端心跳。

预期：

- 心跳失败。
- session 被撤销。

### 13.2 解绑设备

预期：

- 设备状态变更。
- session 被撤销。
- 设备数量可释放。

### 13.3 machine_id 伪造

场景：

- 攻击者使用相同 machine_id 但不同 device key。

预期：

- 如果设备已绑定 public key，请求签名失败。

## 14. 更新包安全测试

### 14.1 上传文件 hash

预期：

- 服务端计算 sha256 正确。

### 14.2 签名验证

预期：

- SDK 使用 public key 验签成功。

### 14.3 文件被篡改

步骤：

1. 下载文件。
2. 修改一个字节。
3. SDK 验证。

预期：

- hash 或签名验证失败。

### 14.4 下载 token 过期

预期：

- 过期 token 无法下载。

### 14.5 下载 token 文件不匹配

预期：

- token 绑定 A 文件时不能下载 B 文件。

### 14.6 路径穿越

请求：

```text
/api/client/releases/download/../../config.yaml
```

预期：

- 返回 400。
- 不读取任意文件。

### 14.7 防回滚

场景：

- 客户端已安装 version_code 100。
- 服务端返回 version_code 99。

预期：

- SDK 拒绝安装，除非有明确 rollback 策略。

## 15. 脚本安全测试

### 15.1 draft 脚本不可拉取

预期：

- 客户端请求失败。

### 15.2 无 feature 权限

预期：

- 拉取失败。

### 15.3 脚本内容篡改

预期：

- SDK 验签失败。

### 15.4 过期脚本

预期：

- 不能拉取或不能执行。

## 16. 审计日志测试

### 16.1 关键操作审计

必须产生审计：

- 登录。
- 创建 app。
- 创建 license。
- 吊销 license。
- 拉黑 device。
- 上传 release。
- 发布 script。
- 修改权限。

### 16.2 敏感字段脱敏

请求体包含：

```json
{
  "password": "secret",
  "refresh_token": "rt",
  "app_secret": "as"
}
```

预期：

- 审计日志中显示为 `***`。

## 17. 文件上传测试

### 17.1 超大文件

预期：

- 超过限制返回 413 或 400。
- 不写入完整临时文件。

### 17.2 非法文件名

预期：

- 过滤或拒绝。

### 17.3 流式处理

预期：

- 大文件上传不会导致内存暴涨。

## 18. API 输入校验测试

必须测试：

- 缺少字段。
- 字段类型错误。
- 超长字符串。
- 非法 UUID。
- 非法时间。
- 非法 enum。
- JSON body 过大。

预期：

- 返回 400。
- 不产生数据库变更。

## 19. 数据库约束测试

必须验证：

- 同租户 email 唯一。
- 不同租户 email 可重复。
- app_key 全局唯一。
- license_key_hash 唯一。
- devices 同 app 下 machine_id 唯一。

## 20. 部署安全检查

生产环境启动前必须检查：

- `APP_ENV=production`
- HTTPS 已启用或由反向代理保证。
- `APP_BASE_URL` 使用 HTTPS 地址。
- `DATABASE_URL` 不为空。
- Redis 密码不为空。
- Cookie secure 开启。
- CORS allowed origins 非空。
- session secret 长度足够。
- token hash pepper 长度足够且不是示例值。
- refresh token pepper 长度足够且不是示例值。
- csrf secret 长度足够且不是示例值。
- master key 存在。

## 21. 依赖安全测试

每次发布前运行：

```text
cargo audit
cargo deny
npm audit
```

如果使用 Go：

```text
govulncheck ./...
```

## 22. 最低上线安全门槛

未满足以下条件，不允许上线：

- 后台不使用 localStorage token。
- 后台 session 可撤销。
- refresh token 轮换。
- 客户端 session 可撤销。
- 跨租户测试通过。
- 权限测试通过。
- 文件下载防路径穿越通过。
- 文件签名验证通过。
- 审计脱敏通过。
- 登录限流通过。
- HTTPS 配置完成。
