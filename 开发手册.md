# 后台系统开发手册

本文档用于指导 AI 从零开发一套完整的后台系统。目标不是普通用户管理后台，而是一个包含用户、租户、权限、客户、软件授权、设备管理、客户端会话、热更新、脚本下发、审计和安全控制的一体化平台。

开发时必须把本文档当作约束文档使用。未在本文档中明确的内容，优先按安全、稳定、可维护、最小可用原则处理。

## 1. 项目目标

系统要解决的问题：

- 管理后台用户可以安全登录、管理团队和权限。
- 平台支持多租户，每个租户的数据严格隔离。
- 租户可以创建自己的软件应用。
- 租户可以给客户发放授权码或订阅。
- 客户端软件可以激活、登录、验证授权、心跳续期。
- 系统可以绑定设备、限制设备数量、拉黑异常设备。
- 系统可以安全分发版本包、热更新包和脚本。
- 客户端必须验证下载内容的完整性和签名。
- 所有关键操作必须有审计日志。
- 系统必须为 AI 继续开发留出清晰边界和模块结构。

一句话目标：

```text
做一个完整的用户管理 + 软件授权 + 客户端安全控制平台。
```

## 2. 非目标

第一版不要做以下内容：

- 不做完整 OIDC / OAuth2 身份提供商。
- 不做 SSO。
- 不做微服务拆分。
- 不做复杂插件市场。
- 不做复杂财务系统。
- 不做过度抽象的权限表达式引擎。
- 不做多语言 SDK 全量覆盖。
- 不做无法验证的“防破解承诺”。

这些可以后续演进，但第一版重点是跑通业务闭环。

## 3. 总体架构

第一版采用模块化单体架构。

```text
React 管理后台
        |
        v
Backend API
        |
        +-- 身份模块
        +-- 租户模块
        +-- 权限模块
        +-- 客户模块
        +-- 应用模块
        +-- 授权模块
        +-- 订阅模块
        +-- 设备模块
        +-- 客户端会话模块
        +-- 更新分发模块
        +-- 脚本分发模块
        +-- 审计模块
        +-- 系统配置模块
        |
        +-- PostgreSQL
        +-- Redis
        +-- S3 / MinIO
```

第一版只有一个后端服务、一个数据库、一个管理后台。内部按模块拆分，部署上保持简单。

## 6. 多租户设计

所有业务数据必须属于某个租户。

核心规则：

- 所有业务表必须有 `tenant_id`。
- 所有查询必须带 `tenant_id` 条件。
- 不允许前端传来的 `tenant_id` 直接作为可信值。
- 后端必须从登录 session 或客户端 session 中取 `tenant_id`。
- 管理员也不能跨租户访问数据，除非未来明确加入平台超级管理员功能。

禁止写法：

```sql
SELECT * FROM licenses WHERE id = $1;
```

必须写成：

```sql
SELECT * FROM licenses WHERE id = $1 AND tenant_id = $2;
```

## 7. 身份与登录设计

后台用户和软件客户必须分开。

后台用户表：

```text
team_members
```

软件客户表：

```text
customers
```

后台用户用于管理系统，软件客户用于客户端登录、订阅和授权。

### 7.1 后台登录

后台登录必须使用服务端 session，不允许前端长期持有 access token。

推荐流程：

```text
1. 用户输入邮箱和密码
2. 后端校验密码和 MFA
3. 创建 admin_session
4. 创建 refresh_token 记录
5. 返回 HttpOnly Cookie
6. 前端只保存用户展示信息，不保存 token
```

Cookie 必须设置：

```text
HttpOnly
Secure
SameSite=Lax 或 Strict
Path=/
```

如果后台前后端同域，优先使用 SameSite=Lax。跨域部署时必须严格配置 CORS 和 CSRF。

### 7.2 密码策略

密码哈希必须使用：

```text
Argon2id
```

最低要求：

```text
最小长度 10
必须包含字母
必须包含数字
必须包含特殊字符
禁止常见弱密码
```

密码相关接口必须限流：

- 登录
- 注册
- 忘记密码
- 重置密码
- MFA 验证

### 7.3 MFA

管理员必须支持 MFA。

第一版支持：

```text
TOTP
恢复码
```

恢复码要求：

- 只展示一次。
- 数据库只存 hash。
- 使用后立即失效。
- 可以重新生成，但重新生成会废弃旧恢复码。

### 7.4 Session 设计

后台 session 表建议：

```text
admin_sessions
  id
  tenant_id
  team_member_id
  user_agent
  ip
  created_at
  last_seen_at
  expires_at
  revoked_at
```

refresh token 表建议：

```text
admin_refresh_tokens
  id
  session_id
  token_hash
  expires_at
  used_at
  revoked_at
  created_at
```

refresh token 必须轮换。

规则：

- 每次 refresh 都签发新 refresh token。
- 旧 refresh token 标记 used。
- 如果已 used 的 refresh token 再次出现，判定为疑似泄露，撤销整个 session。

## 8. 权限设计

第一版采用 RBAC。

内置角色：

```text

admin
developer

```

权限格式：

```text

```

示例：

```text

member:update
app:create
app:update

```

权限判断必须在后端执行，前端隐藏菜单只作为体验优化。

禁止：

- 前端传 role 后端相信。
- 只靠路由名称判断权限。
- 不带 tenant_id 查资源。

## 14. 设备密钥设计

为了提高客户端安全性，建议使用设备密钥对。

首次激活流程：

```text
1. 客户端生成 device keypair
2. 客户端提交 public key
3. 服务端绑定 public key 到 device
4. 后续请求使用 private key 签名
5. 服务端使用 public key 验签
```

设备密钥表：

```text
device_keys
  id
  tenant_id
  app_id
  device_id
  public_key
  algorithm
  status
  created_at
  rotated_at
  revoked_at
```

推荐算法：

```text
Ed25519
```

`device_public_key` 必须是可解析的 Ed25519 public key，支持 PEM 或 base64/base64url 原始 32 字节公钥。服务端写入 `device_keys` 前必须校验格式。

设备 key 支持客户端自助轮换：

```text
POST /api/client/devices/self/rotate-key
```

轮换请求必须使用当前 active key 签名。服务端按 `X-Device-Key-Id` 精确把旧 key 标记为 `rotated`，再插入新的 active key，避免并发轮换误操作刚创建的新 key。轮换不会撤销当前 session，客户端成功后必须保存新私钥和新 `device_key_id`。

请求签名内容：

```text
method
path
body_sha256
timestamp
nonce
device_id
device_key_id
session_id
```

服务端必须检查：

- timestamp 在允许窗口内。
- nonce 没用过。
- device 未被禁用。
- session 未撤销。
- 签名有效。

nonce 存 Redis，TTL 建议 5 分钟。

nonce 必须是 16–128 个 URL-safe 字符：`A-Z`、`a-z`、`0-9`、`-`、`_`。

## 16. 客户端接口设计

基础接口：

```text
POST /api/client/auth/activate
POST /api/client/auth/login
POST /api/client/auth/refresh
POST /api/client/auth/logout
POST /api/client/auth/heartbeat
POST /api/client/auth/verify
DELETE /api/client/devices/self
POST /api/client/devices/self/rotate-key
```

激活返回：

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "token_type": "Bearer",
  "expires_in": 900,
  "refresh_expires_in": 2592000,
  "session_id": "...",
  "device_id": "...",
  "device_key_id": "...",
  "subscription_id": null,
  "entitlement_id": "...",
  "entitlement_kind": "license",
  "entitlement_status": "active",
  "entitlement_active": true,
  "features": []
}
```

客户账号登录不再要求必须存在有效订阅；无有效订阅时仍返回 session，但 `subscription_id=null`、`entitlement_active=false`、`features=[]`。需要订阅的业务接口必须在各自入口单独校验，AI 客户端接口必须强制有效订阅。

客户端认证接口必须有独立限流。

## 17. 请求防重放

客户端敏感请求必须支持签名。

请求头建议：

```text
X-Device-Id
X-Device-Key-Id
X-Timestamp
X-Nonce
X-Body-SHA256
X-Signature
```

服务端验证顺序：

```text
1. 检查 timestamp 是否过期
2. 检查 nonce 是否重复
3. 检查 device 是否存在且 active
4. 检查 session 是否有效
5. 用 X-Device-Key-Id 找到 active device public key 验签
6. 执行业务逻辑
```

注意：

- 不要只依赖 machine_id。
- 不要只依赖 app_secret。
- 不要相信客户端传来的设备信息。

## 18. 更新包分发

版本表：

```text
releases
  id
  tenant_id
  app_id
  version
  version_code
  status
  changelog
  force_update
  file_id
  signing_key_id
  signature_kid
  signature
  signature_alg
  created_at
  published_at
```

文件表：

```text
release_files
  id
  tenant_id
  app_id
  storage_key
  file_name
  file_size
  sha256
  signature
  signature_alg
  created_at
```

上传流程：

```text
1. 后台上传文件
2. 服务端流式保存到对象存储
3. 同时计算 sha256
4. 服务端用应用私钥签名 sha256 + file_size
5. 保存签名和文件元数据
```

发布流程：

```text
1. 后台发布 draft release
2. 服务端写入确定的 published_at
3. 服务端签名 app_id/version/version_code/sha256/file_size/published_at_unix
4. 保存 release 元数据签名
```

下载流程：

```text
1. 客户端请求最新版本
2. 服务端检查授权/订阅/设备
3. 服务端返回版本信息、hash、release 元数据签名、短时下载 URL
4. 客户端下载
5. 客户端校验 hash
6. 客户端校验 release 元数据签名
7. 通过后安装
```

短时下载 token：

```text
有效期 5 分钟
绑定 app_id、device_id、file_id、filename
只用于下载
不能用于 API 认证
```

## 19. 防回滚设计

更新元数据签名必须包含：

```text
app_id
version
version_code
sha256
file_size
published_at_unix
```

客户端必须记录已安装最高 `version_code`。

如果服务端返回更低版本，除非明确 rollback 策略，否则拒绝安装。

## 20. 脚本下发

脚本表：

```text
secure_scripts
  id
  tenant_id
  app_id
  name
  version
  status
  content_ciphertext
  content_sha256
  signature
  signature_alg
  required_features_json
  expires_at
  created_at
  published_at
```

脚本状态：

```text
draft
published
deprecated
revoked
```

脚本下发必须检查：

- app 是否有效。
- device 是否有效。
- session 是否有效。
- 授权/订阅是否具备 required features。
- 脚本是否 published。
- 脚本是否过期。

客户端必须验签后再执行。

脚本内容处理规则：

- 服务端使用 `content_ciphertext` 加密存储脚本内容。
- 客户端拉取时，服务端解密后返回 `content_base64`。
- 客户端不负责解密，只负责验证 hash 和签名。
- 如果未来需要端到端加密脚本，必须单独设计客户端密钥分发。

## 21. 签名规范

版本文件登记签名 payload：

```text
sha256 + ":" + file_size
```

release 元数据签名 payload：

```text
app_id
version
version_code
sha256
file_size
published_at_unix
```

release 元数据字段按上面的顺序使用 `\n` 拼接。签名前必须使用稳定序列化格式。

推荐 JSON canonicalization 或明确拼接格式。

不要使用普通 JSON 直接 marshal 后签名，避免字段顺序和序列化差异造成 SDK 验签问题。

## 23. 限流策略

必须使用 Redis 做分布式限流。

建议：

```text
后台登录: 按 IP + email 限流
客户登录: 按 IP + email 限流
客户端激活: 按 IP + app_key + machine_id 限流
refresh: 按 session_id + IP 限流
heartbeat: 按 device_id 限流
下载: 按 file_id + device_id 限流
```

不要只用内存限流。内存限流在多实例部署下无效。

## 24. 安全响应头

生产环境必须开启：

```text
Strict-Transport-Security
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy
Permissions-Policy
Content-Security-Policy
```

CSP 第一版建议：

```text
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
img-src 'self' data: https:;
connect-src 'self';
frame-ancestors 'none';
```

禁止：

```text
script-src 'unsafe-eval'
```

只有明确必要时才允许 `unsafe-inline`，并尽量使用 nonce。

## 25. CORS 规则

生产环境必须配置明确来源。

禁止：

```text
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

允许：

```text
https://admin.example.com
https://api.example.com
```

本地开发来源只能在开发配置里出现。

## 26. CSRF 规则

后台使用 Cookie session 时，必须启用 CSRF。

建议：

```text
双提交 Cookie
或服务端生成 CSRF token
```

所有非安全方法都要验证：

```text
POST
PUT
PATCH
DELETE
```

客户端 SDK 使用 Bearer token 时，一般不需要 CSRF，但必须有签名、防重放和限流。

## 27. 密钥管理

密钥类型：

```text
数据库密码
Redis 密码
session 签名密钥
refresh token pepper
一次性 token / 下载 token / 授权码 hash pepper
应用签名私钥
JWT access token 签名私钥
更新包签名私钥
脚本签名私钥
对象存储密钥
邮件 SMTP 密码
```

规则：

- 不要提交真实密钥。
- `.env.example` 只能放占位符。
- 生产密钥必须来自环境变量或密钥管理系统。
- 私钥必须加密存储。
- 密钥轮换必须有方案。

应用私钥加密建议：

```text
AES-256-GCM
主密钥来自 MASTER_KEY
```

统一要求：

- 全系统统一使用 `MASTER_KEY` 作为应用层加密主密钥名称，不再使用 `LICENSE_MASTER_KEY`。
- `MASTER_KEY` 只用于加密私钥或敏感配置，不直接用于业务签名。
- `MASTER_KEY` 丢失后，被它加密的数据无法恢复，生产环境必须离线备份。

签名密钥要求：

```text
signing_keys
  key_scope
  kid
  alg
  public_key_pem
  private_key_envelope
  status
```

`key_scope` 至少包含：

```text
jwt_access_token
release_file
secure_script
app_request
```

JWKS 要求：

- 所有对外可验证的签名都必须带 `kid`。
- 服务端必须提供 JWKS 公钥接口，只返回 public key。
- 客户端 access token 必须使用全局 `jwt_access_token` Ed25519 key 签发，JWT Header 必须包含 `kid`。
- access token、更新包、脚本签名都要能通过 `kid` 找到对应公钥。
- 轮换密钥时，旧 key 不能马上删除，要保留到旧 token、旧更新包、旧脚本过期。
- `revoked` key 不能再用于新签名。

全局 JWT access token key 轮换：

- 后台使用 `POST /api/admin/security/jwt-signing-keys/rotate` 主动轮换全局 `jwt_access_token` key。
- 接口必须要求 `security:rotate_key` 权限，并写入 `jwt.signing_key.rotate` 审计。
- 新 key 创建后立即用于签发 access token，旧 active key 改为 `retiring`。
- 旧 key 必须写入 `not_after = now + CLIENT_ACCESS_TOKEN_TTL_SECONDS`。
- `/.well-known/jwks.json` 必须在旧 key 的 `not_after` 前继续返回 `active` 和 `retiring` 的全局 `jwt_access_token` 公钥。

## 28. 数据库设计原则

必须使用 PostgreSQL。

重要索引：

## 29. 事务规则

这些操作必须放在事务里：

- 创建租户 + 创建 owner。
- 激活授权 + 创建设备 + 创建 session。
- 客户登录 + 创建设备 + 创建 session。
- 吊销授权 + 撤销相关设备 session。
- 拉黑设备 + 撤销相关 session。
- 刷新 token + 轮换 refresh token。
- 发布版本 + 写文件元数据。

不要在事务中执行慢操作：

- 文件上传。
- 大文件 hash。
- 外部 HTTP 请求。
- 发邮件。

这些要拆成任务或 outbox。

## 30. 后台任务

需要 worker 处理：

- 清理过期 session。
- 清理过期 refresh token。
- 清理过期下载 token。
- 清理未发布的临时上传文件。
- 发送邮件。
- 处理 webhook。
- 异步写统计。
- 对账和完整性检查。

任务必须幂等。

## 31. API 返回格式

统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_xxx"
}
```

错误返回：

```json
{
  "code": 40100,
  "message": "unauthenticated",
  "data": null,
  "request_id": "req_xxx"
}
```

错误码必须稳定，不能随意改。详细错误码以 `权限点与错误码清单.md` 为准。

## 32. 错误处理

禁止把内部错误直接返回前端。

错误分类：

```text
400 参数错误
401 未认证
403 无权限
404 资源不存在
409 状态冲突
429 限流
500 服务异常
```

日志中记录内部错误，返回给用户的信息要克制。

## 33. 前端安全规则

前端禁止：

- localStorage 保存后台 token。
- sessionStorage 保存后台 token。
- 在 URL 中传后台 token。
- 打印 token 到 console。
- 在错误提示中展示敏感信息。

前端可以保存：

```text
当前用户展示信息
菜单状态
主题设置
```

所有权限判断以后端为准。

## 34. SDK 设计

第一版至少做一个 SDK。

建议先做：

```text
Go SDK 或 Rust SDK
```

SDK 必须提供：

- 激活。
- 登录。
- 刷新 session。
- 心跳。
- 验证授权。
- 下载更新。
- 校验 hash。
- 校验签名。
- 本地缓存。
- 注销。

SDK 本地缓存必须加密。

缓存内容包括：

```text
license info
device id
session id
access token
refresh token
expires_at
```

refresh token 存客户端有风险，但客户端必须使用时只能尽量保护：

- 存 OS keychain 优先。
- 否则使用机器相关信息派生密钥加密。
- 缓存文件权限收紧。
- logout 清理缓存。

## 35. 客户端安全边界

必须明确：

客户端不可信。

不要依赖：

- 客户端传来的时间。
- 客户端传来的设备名。
- 客户端传来的授权状态。
- 客户端本地缓存中的有效期。
- 客户端隐藏的 app_secret。

服务端必须始终重新判断：

- 授权是否有效。
- 订阅是否有效。
- 设备是否有效。
- session 是否有效。
- 请求是否重放。
- 签名是否正确。

## 36. 配置文件

配置示例：

```yaml
server:
  host: "0.0.0.0"
  port: 8080
  mode: "release"

database:
  url: "${DATABASE_URL}"

redis:
  url: "${REDIS_URL}"

security:
  cookie_secure: true
  allowed_origins:
    - "https://admin.example.com"
  access_token_ttl_seconds: 900
  refresh_token_ttl_seconds: 2592000
  download_token_ttl_seconds: 300
```

生产环境启动时必须校验：

- mode 是否 release。
- DATABASE_URL 是否存在。
- Redis 是否存在。
- allowed_origins 是否配置。
- session 密钥长度是否足够。
- `ADMIN_SESSION_TTL_SECONDS`、`CLIENT_ACCESS_TOKEN_TTL_SECONDS`、`CLIENT_REFRESH_TOKEN_TTL_SECONDS`、`CLIENT_SESSION_TTL_SECONDS`、`DOWNLOAD_TOKEN_TTL_SECONDS` 必须大于 0。
- HTTPS 是否由网关保证。

## 40. AI 开发规则

给 AI 开发时必须遵守：

1. 每次只实现一个模块或一个小闭环。
2. 改代码前先读相关代码。
3. 不允许顺手重构无关模块。
4. 所有数据库变更必须写 migration。
5. 所有新增接口必须补权限检查。
6. 所有业务查询必须带 tenant_id。
7. 所有敏感字段不能返回前端。
8. 所有 token 只能返回一次。
9. 所有密钥不能写死。
10. 所有安全逻辑必须有测试。

AI 每次完成任务必须输出：

```text
改了哪些文件
新增哪些接口
新增哪些表
安全影响
验证命令
剩余风险
```

## 41. 禁止事项

严禁：

- 后台 token 放 localStorage。
- 明文存密码。
- 明文存 refresh token。
- 明文存 app_secret。
- 直接信任前端 role。
- 查询资源不带 tenant_id。
- 生产 CORS 使用 `*`。
- 生产关闭 HTTPS。
- 日志打印 token。
- 下载接口允许 `../`。
- 文件下载不校验授权。
- 客户端只靠 machine_id 判断授权。
- refresh token 不轮换。
- 改密码后旧 session 继续长期有效。
- 管理员没有 MFA。

## 45. 关键验收标准

安全验收：

- 所有后台状态修改接口都有 CSRF。
- 后台没有 localStorage token。
- refresh token 复用会撤销 session。
- 跨租户访问失败。
- 低权限越权失败。
- 文件下载必须带有效 token 或有效客户端 session。
- 下载文件必须验签。
- 审计日志不泄露敏感信息。

稳定性验收：

- 后端重启后 session 状态不丢。
- Redis 短暂不可用时给出明确错误。
- 数据库迁移可重复执行。
- worker 任务幂等。
- 备份可以恢复。

性能验收：

- 登录接口有独立限流。
- 心跳接口可承载高频请求。
- 文件上传下载不整包读内存。
- 高频查询有索引。
- 审计日志不会拖慢主流程。

## 46. 最终设计原则

系统的最终原则：

```text
身份和授权分清楚
后台用户和客户分清楚
客户端永远不可信
所有关键判断放服务端
所有 token 可撤销
所有文件可验签
所有资源有租户边界
所有关键操作有审计
```

只要后续 AI 开发严格遵守这些原则，系统可以逐步做大，而不会变成安全边界混乱的后台项目。
