# 文档完整性检查

检查时间：2026-06-02

检查目标：

- 确认每份文档是否覆盖当前系统方向。
- 找出“概念提到了，但数据库/API/任务没有落地”的缺口。
- 修正命名不一致、验收缺失、AI 开发容易漏做的问题。

结论：

当前文档主线完整，可以作为从零开发自有用户管理后台系统的开发依据。检查中发现的关键缺口已经补齐，主要集中在权限错误码、JWKS/签名密钥、一次性 token、MFA 恢复码、环境变量命名一致性。

第二次复查补充：

- API 中部分敏感操作权限过粗，已改为更精确的权限点。
- `审计详情` 接口缺少权限说明，已补 `audit:read`。
- 开发手册提到 MFA 恢复码可重新生成，但 API 缺少接口，已补恢复码重新生成接口。
- `email_verified` 和 `email_verify` token 有字段和表，但 API 缺少邮箱验证流程，已补后台成员和客户邮箱验证接口。
- 环境变量里 `APP_ENV` 和 `SERVER_MODE` 双命名，已统一保留 `APP_ENV`。

第三次复查优化：

- API 失败示例错误码和 message 不一致，已改为 `40100 unauthenticated`，并统一返回 `request_id`。
- API 文档“所有接口均以 /api 为前缀”与 JWKS 标准路径冲突，已改为“业务接口以 /api 为前缀”。
- 客户端签名“第一版可选”表述偏弱，已改为 MVP 关键接口必须签名、生产前保护接口全部签名。
- `signing_keys.private_key_ciphertext` 表达不够完整，已改为 `private_key_envelope`，明确保存 ciphertext、nonce、tag、key_version。
- `device_keys` 增加 `tenant_id`、`app_id`，并明确 `X-Device-Key-Id` 对应 `device_keys.id`。
- 更新包和脚本的 `signing_key_id` 改为 not null。
- 新增 `TOKEN_HASH_PEPPER`，用于一次性 token、下载 token、授权码 hash。
- 脚本下发去掉客户端“解密内容”的歧义，改为服务端加密存储、下发 `content_base64`、客户端只做 hash 和签名校验。

用途：客户端安全通信、授权、更新、脚本拉取的 SDK 设计。

- 错误类型。
- 重试。
- 日志。
- 缓存。
- logout。
- 解绑设备。
- 公共 API。

发现问题：

- 文件和脚本验签写了签名校验，但没有明确从哪里获取公钥。
- 签名轮换时缺少 `kid` 缓存和刷新规则。

已补充：

- 新增 JWKS 公钥缓存规则。
- 明确 SDK 按 `signature_kid` 查本地公钥。
- 找不到 `kid` 时刷新 JWKS。
- 刷新后仍找不到则拒绝安装更新包或使用脚本。
- 缓存结构增加 `jwks_cache`。
- 明确 SDK 第一版不负责脚本端到端解密，只负责 `content_base64` 解码、hash 校验和签名校验。

后续注意：

- SDK 不能因为网络失败就跳过验签。
- JWKS 缓存过期且无法刷新时，禁止安装新更新包和新脚本。
- 客户端私钥必须只保存在本地安全存储。
