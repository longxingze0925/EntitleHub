# 部署运维手册

本文档用于指导后台系统从开发环境部署到生产环境。重点覆盖部署架构、环境变量、HTTPS、数据库、Redis、对象存储、备份、监控、日志、发布、回滚和安全检查。

## 1. 部署目标

生产部署必须满足：

- 全站 HTTPS。
- 数据库不暴露公网。
- Redis 不暴露公网。
- 对象存储私有。
- 后台 API 可水平扩展。
- 有健康检查。
- 有日志。
- 有监控。
- 有备份。
- 有回滚方案。
- 有安全启动检查。

## 2. 推荐部署架构

第一阶段使用 Docker Compose。

```text
Internet
   |
   v
Caddy / Nginx
   |
   +-- admin frontend
   +-- backend api
        |
        +-- PostgreSQL
        +-- Redis
        +-- MinIO
```

组件：

```text
caddy/nginx: HTTPS、反向代理、静态文件
backend: Rust/Go API
admin: React 构建产物
postgres: 主数据库
redis: session cache、限流、nonce
minio: 文件存储
```

第二阶段再考虑：

```text
Kubernetes
Managed PostgreSQL
Managed Redis
S3
CDN
```

## 3. 域名规划

建议：

```text
admin.example.com  管理后台
api.example.com    API
files.example.com  文件下载，可选
```

如果前后端同域：

```text
example.com
example.com/api
```

同域更适合 Cookie session 和 CSRF，复杂度较低。

## 4. HTTPS 要求

生产环境必须：

- 只开放 HTTPS。
- HTTP 自动跳转 HTTPS。
- 开启 HSTS。
- 使用 TLS 1.2/1.3。
- 禁止弱加密套件。

Caddy 推荐配置：

```text
admin.example.com {
  encode gzip
  root * /var/www/admin
  try_files {path} /index.html
  file_server
}

api.example.com {
  reverse_proxy backend:8080
}
```

Nginx 反代时必须传递：

```text
X-Forwarded-For
X-Forwarded-Proto
X-Real-IP
Host
```

后端必须只信任配置中的 trusted proxies。

## 6. 密钥生成

建议：

```bash
openssl rand -base64 32
```

密钥类型：

```text
SESSION_SECRET
CSRF_SECRET
REFRESH_TOKEN_PEPPER
TOKEN_HASH_PEPPER
MASTER_KEY
OBJECT_STORAGE_SECRET_KEY
SMTP_PASSWORD
ALERTMANAGER_WEBHOOK_TOKEN
```

`MASTER_KEY` 用于加密应用私钥、签名私钥和敏感配置，丢失后加密数据无法恢复。必须备份。底层对象存储可以是本地磁盘、S3 或 MinIO，但环境变量统一使用 `OBJECT_STORAGE_*`。`OBJECT_STORAGE_MODE=local` 时必须配置 `OBJECT_STORAGE_LOCAL_ROOT`；`OBJECT_STORAGE_MODE=s3` 时必须配置 `OBJECT_STORAGE_ENDPOINT`、`OBJECT_STORAGE_BUCKET`、`OBJECT_STORAGE_ACCESS_KEY`、`OBJECT_STORAGE_SECRET_KEY`，`OBJECT_STORAGE_REGION` 默认 `us-east-1`。

## 7. 数据库部署

PostgreSQL 要求：

- 使用专用数据库用户。
- 禁止使用 postgres 超级用户跑应用。
- 数据库不暴露公网。
- 开启自动备份。
- 配置连接池。

建议参数：

```text
max_connections 根据实例大小设置
shared_buffers 合理配置
wal_level replica
timezone UTC
```

应用连接池建议：

```text
min_connections=5
max_connections=50
connect_timeout=5s
idle_timeout=300s
```

## 8. Redis 部署

Redis 用途：

- 限流。
- nonce 防重放。
- session 热缓存。
- 短期验证码。

要求：

- 设置密码。
- 不暴露公网。
- 开启持久化可选。
- 限流和 nonce 数据允许丢失，但不能影响核心数据库一致性。

Redis 不作为唯一数据源。

## 9. 对象存储

对象存储用于：

- 版本包。
- 热更新文件。
- 脚本包。
- 临时上传文件。

要求：

- bucket 私有。
- 下载必须通过后端生成短时 URL 或后端代理。
- 文件名必须规范化。
- 存储 key 不直接使用用户上传文件名。

推荐 key 格式：

```text
tenants/{tenant_id}/apps/{app_id}/releases/{file_id}
tenants/{tenant_id}/apps/{app_id}/scripts/{script_id}
```

本地试运行：

```powershell
pwsh -File ops/new-compose-env.ps1
# 如本机已有其他服务占用端口，可传入 -BackendHostPort、-RedisHostPort、-GrafanaHostPort 等参数，或在 .env.compose 中调整。
docker compose -p user-admin --env-file .env.compose up -d postgres redis
```

```bash
cp .env.compose.example .env.compose
openssl rand -base64 32
```

数据库 migration 示例：

```bash
cd backend
DATABASE_URL=postgres://app_user:password@127.0.0.1:5432/user_admin sqlx migrate run
cd ..
docker compose -p user-admin --env-file .env.compose up -d --build
pwsh -File ops/smoke-compose.ps1
```

也可以使用后端容器内置 migration 子命令，避免部署机额外安装 `sqlx-cli`：

```bash
docker compose -p user-admin --env-file .env.compose run --rm backend user-admin-backend migrate
```

首次 owner 初始化和后台登录链路可以用本地 smoke 脚本验证：

```powershell
pwsh -File ops/smoke-init-owner.ps1 -RunMigrations
```

该脚本只会在没有 active tenant 时执行 `init-owner`，随后验证 `/api/auth/login` 和 `/api/auth/me`。自动生成的本地 smoke 凭据保存在 `.tools/init-owner-smoke.env`，该目录已被忽略，不应提交。

本地 compose 栈和 owner 登录验证通过后，可以运行 Client SDK 在线 smoke：

```powershell
pwsh -File ops/smoke-client-sdk.ps1
```

该脚本通过后台 API 复用或创建 `sdk-smoke` 应用，创建短期测试 license，然后运行 SDK ignored live test，覆盖客户端激活、refresh token 刷新、JWKS 验证 access token、签名 heartbeat 和响应解析。脚本不会输出 owner 密码或 license key。

AI 网关配置好真实三方渠道、模型、客户钱包余额和客户 AI API Key 后，可以运行：

```powershell
$env:AI_GATEWAY_API_KEY = "ehai_..."
pwsh -File ops/smoke-ai-gateway.ps1 -ChatModel gpt-4o-mini -EmbeddingModel text-embedding-3-small
```

需要验证图片生成和缓存素材下载时，再追加 `-ImageModel image-test`。图片测试会产生真实三方费用。

backend 健康检查：

```text
GET /health
```

监控入口：

```text
GET /metrics
Prometheus: http://localhost:9090/
Alertmanager: http://localhost:9093/
Grafana: http://localhost:3000/
PostgreSQL exporter: http://localhost:9187/metrics
Redis exporter: http://localhost:9121/metrics
```

这些地址使用默认宿主机端口，且默认只绑定 `127.0.0.1`。若 `.env.compose` 中调整了 `BACKEND_HOST_PORT`、`REDIS_HOST_PORT`、`GRAFANA_HOST_PORT` 等变量，本机浏览器和 CLI 访问应使用调整后的宿主机端口；容器内部仍使用 `backend:8080`、`redis:6379` 等服务端口。只有确实需要局域网访问本机 compose 栈时，才把 `COMPOSE_HOST_BIND` 改成 `0.0.0.0`。

compose 默认使用明确版本的 Prometheus、Alertmanager、Grafana、postgres-exporter 和 redis-exporter 镜像。生产环境必须从部署平台或 secret store 设置 `GRAFANA_ADMIN_PASSWORD`，并把 `PROMETHEUS_IMAGE`、`ALERTMANAGER_IMAGE`、`GRAFANA_IMAGE`、`POSTGRES_EXPORTER_IMAGE`、`REDIS_EXPORTER_IMAGE` 固定到已审核版本或镜像 digest。
平台业务通知渠道在后台 `通知渠道` 页面配置，webhook URL、SMTP password、PagerDuty routing key 等敏感字段会通过 `MASTER_KEY` 加密保存到 `notification_channels.secret_encrypted`，列表和审计日志只显示 `secret_configured`。
通知渠道页面支持两类测试：默认 dry-run 只校验配置和密钥完整性；受控真实测试需要二次确认，会向外部 webhook、SMTP 或 PagerDuty 发送测试消息，并写入审计和 `notification_delivery_*` 指标。
Alertmanager 默认仍使用 no-op receiver。推荐生产路径是让 Alertmanager 调用后端内部入口 `POST /api/internal/alertmanager/webhook`，后端再读取后台页面配置的通知渠道投递；这样 Alertmanager YAML 只需要保存入口鉴权 token，不保存 webhook URL、SMTP password、PagerDuty routing key 等业务通知密钥。

可以用脚本从环境变量渲染 Alertmanager 配置，并在写入前用 `amtool` 校验：

```powershell
pwsh -File ops/render-alertmanager-config.ps1 -Mode backend -Check
pwsh -File ops/render-alertmanager-config.ps1 -Mode webhook -WebhookUrl https://alert-receiver.example.com/prometheus -Check
pwsh -File ops/render-alertmanager-config.ps1 -Mode email -Check
pwsh -File ops/render-alertmanager-config.ps1 -Mode production -Check
```

默认输出到 `.tools/alertmanager.generated.yml`。本地 compose 推荐把 backend 模式配置放在 `.tools/`，并通过 `ALERTMANAGER_CONFIG_PATH` 指向该文件，避免把入口 token 写进仓库内的 `ops/alertmanager/alertmanager.yml`：

```powershell
pwsh -File ops/activate-alertmanager-backend.ps1 -Check -Restart
pwsh -File ops/smoke-alertmanager-backend.ps1
```

生产替换当前配置时可以显式指定输出路径，但应确保该路径由部署平台的 secret store 或受控配置卷管理，不提交明文 token：

```powershell
pwsh -File ops/render-alertmanager-config.ps1 -Mode backend -OutputPath .tools/alertmanager.backend.yml -Check
```

`backend` 模式读取 `ALERTMANAGER_WEBHOOK_TOKEN`，默认 URL 是 `http://backend:8080/api/internal/alertmanager/webhook`，如部署拓扑不同可设置 `ALERTMANAGER_BACKEND_WEBHOOK_URL`。`webhook`、`email` 和 `production` 模式保留给需要 Alertmanager 直接投递外部系统的场景，应由部署平台注入 `ALERTMANAGER_SMTP_*`、`ALERTMANAGER_EMAIL_TO`、`ALERTMANAGER_CRITICAL_*`、`ALERTMANAGER_WARNING_EMAIL_TO` 等变量；不要把后台页面录入的业务通知密钥再写回 YAML。

镜像 pin 可以用脚本检查：

```powershell
pwsh -File ops/check-compose-image-pins.ps1
pwsh -File ops/check-compose-image-pins.ps1 -RequireDigest
```

默认模式禁止无 tag 和 `latest`；`-RequireDigest` 用于生产发布审核，要求外部镜像使用 `@sha256:` digest。

compose healthcheck 已覆盖 PostgreSQL、Redis、Backend、Admin、Prometheus、Alertmanager、Grafana 和 postgres-exporter。redis-exporter 镜像不包含 shell，在线可用性通过 `ops/smoke-compose.ps1` 的 Prometheus targets 检查覆盖。

发布前建议运行本地 CI 式校验：

```powershell
pwsh -File ops/validate-ci.ps1
```

校验内容包括 OpenAPI `$ref`、监控 YAML/JSON、后端测试、Client SDK 测试、Admin lint/build、Docker Compose config；如果部署机安装了 `promtool` 和 `amtool`，脚本会额外做 Prometheus/Alertmanager 语义级配置校验。
本地 compose 栈启动后，可以运行 `pwsh -File ops/smoke-compose.ps1` 做在线 smoke check，覆盖 Backend health/readiness/metrics、Admin、Prometheus、Alertmanager、Grafana 和 Prometheus targets。
需要验证 Client SDK 真实接口链路时，运行 `pwsh -File ops/smoke-client-sdk.ps1`，覆盖 activation、refresh、JWKS token 校验和 heartbeat。
GitHub Actions 已接入 `.github/workflows/ci.yml`，会复用 `ops/validate-ci.ps1` 并通过 Prometheus/Alertmanager 容器执行语义级配置校验。
本机需要安装 `promtool` 和 `amtool` 时，可以运行：

```powershell
pwsh -File ops/install-monitoring-tools.ps1
pwsh -File ops/validate-ci.ps1 -StrictExternalTools
```

健康检查必须验证：

- API 进程存活。
- 数据库连接可用。
- Redis 可选检查。

## 11. 数据库迁移

发布流程中 migration 必须明确执行。

推荐：

```text
1. 备份数据库
2. 执行 migration
3. 启动新版本
4. 健康检查
5. 验证核心接口
```

禁止：

- 应用每次启动自动执行破坏性 migration。
- 未备份直接执行大 migration。
- migration 中删除列不留兼容期。

安全迁移原则：

```text
先加字段
双写或兼容读取
迁移数据
切换代码
最后删除旧字段
```

```text

cargo fmt --check
cargo audit
npm run build
npm run lint
```

## 13. 回滚流程

回滚分两类：

### 13.1 代码回滚

适用于未改数据库或数据库向后兼容。

```text
1. 切回旧镜像
2. 重启服务
3. 健康检查
```

### 13.2 数据库回滚

高风险。优先避免。

要求：

- 有发布前备份。
- 明确数据损失范围。
- 停止写入。
- 恢复备份。
- 验证数据。

设计 migration 时应尽量避免需要数据库回滚。

## 14. 备份策略

必须备份：

- PostgreSQL。
- 对象存储。
- MASTER_KEY。
- 环境变量。

PostgreSQL：

```text
每日全量备份
WAL/PITR 持续归档
保留 7-30 天
```

对象存储：

```text
开启版本控制
定期同步到异地
```

恢复演练：

```text
每月至少一次
```

当前仓库已提供 PostgreSQL 备份/恢复脚本：

```bash
DATABASE_URL=postgres://app_user:password@127.0.0.1:5432/user_admin ./ops/backup-postgres.sh
DATABASE_URL=postgres://app_user:password@127.0.0.1:5432/user_admin BACKUP_FILE=./backups/user_admin_20260605T120000Z.dump ./ops/restore-postgres.sh
```

本地 compose 栈启动后，可以运行一次不触碰业务库的备份/恢复演练：

```powershell
pwsh -File ops/drill-postgres-backup-restore.ps1
```

该脚本会创建临时源库和临时恢复库，执行 `pg_dump` / `pg_restore`，对比恢复后的数据指纹，并将演练 dump 与 SHA256 写入 `backups/drills`。默认会删除临时数据库。

本地 compose 对象存储 volume 可以运行备份/恢复演练：

```powershell
pwsh -File ops/drill-object-storage-backup-restore.ps1
```

该脚本会把 `object-storage` volume 归档，恢复到一次性 Docker volume，并通过文件 SHA256 清单对比确认恢复结果。演练归档和 SHA256 写入 `backups/object-storage-drills`，默认删除一次性恢复 volume。

`MASTER_KEY` 不应被脚本复制到普通备份目录。可以先校验当前值并生成非明文指纹：

```powershell
pwsh -File ops/check-master-key-backup.ps1
```

如果已经有离线备份副本，可以用指纹确认它和当前 `.env.compose` 中的 `MASTER_KEY` 一致：

```powershell
pwsh -File ops/check-master-key-backup.ps1 -BackupKeyFile D:\offline\MASTER_KEY.txt
```

生产环境仍必须把对象存储和 `MASTER_KEY` 放入部署平台认可的独立备份体系，例如 S3/MinIO 版本控制、跨区域复制、KMS/secret store 导出和离线保管。

## 15. 日志规范

```text

request_id
tenant_id

method
```

- password
- token
- refresh_token
- app_secret
- private_key
- authorization header

以下依赖深度指标由 PostgreSQL/Redis exporter 提供。

必须监控：

```text
postgres_up
redis_up
redis_connected_clients
```

告警建议：

- 5xx 错误率超过 1%。
- P95 延迟超过 1 秒。
- 数据库连接池耗尽。
- Redis 不可用。
- 通知渠道投递失败。
- 登录失败异常增高。
- nonce 重放异常增高。
- refresh token 复用出现。

## 17. 安全启动检查

应用启动时检查：

- `APP_ENV=production`
- `APP_BASE_URL` 已配置为 HTTPS 地址。
- `COOKIE_SECURE=true`
- `ALLOWED_ORIGINS` 非空。
- `SESSION_SECRET` 长度足够。
- `TOKEN_HASH_PEPPER` 长度足够。
- `REFRESH_TOKEN_PEPPER` 长度足够。
- `CSRF_SECRET` 长度足够。
- 生产密钥不使用示例值。
- `MASTER_KEY` 存在。
- 数据库不是默认密码。
- Redis 不是空密码。

不满足直接启动失败。

## 18. 管理后台安全

生产后台必须：

- 不使用 localStorage 存 token。
- 使用 HttpOnly Cookie。
- 开启 CSRF。
- 开启 MFA。
- 开启安全响应头。
- 限制后台域名。

管理后台建议加 IP allowlist，但不是必须。

## 19. 客户端 API 防护

必须：

- HTTPS。
- access token 短有效期。
- refresh token 轮换。
- device session 可撤销。
- 高频接口限流。
- nonce 防重放。
- 请求签名。

高频接口：

```text
/api/client/auth/heartbeat
/api/client/auth/verify
/api/client/auth/refresh
```

这些接口必须单独指标和限流。

## 20. 故障处理

### 20.1 数据库不可用

表现：

- API 500。
- 健康检查失败。

处理：

1. 检查数据库进程。
2. 检查连接数。
3. 检查慢查询。
4. 必要时重启 backend。
5. 仍不可用时切换备份或恢复。

### 20.2 Redis 不可用

影响：

- 限流失效或全部拒绝，取决于策略。
- nonce 防重放不可用。
- session 热缓存不可用。

建议策略：

```text
认证和签名相关 Redis 不可用时 fail closed
普通缓存 Redis 不可用时 fail open
```

### 20.3 对象存储不可用

影响：

- 上传失败。
- 下载失败。

处理：

- 不影响登录和授权验证。
- 告警并降级隐藏上传入口。

## 21. 性能压测

必须压测：

- 后台登录。
- 客户端激活。
- 客户端 refresh。
- 客户端 heartbeat。
- 最新版本查询。
- 文件下载。

最低目标：

```text
heartbeat 500 RPS
refresh 100 RPS
后台普通接口 P95 < 300ms
客户端认证接口 P95 < 500ms
```

实际目标根据业务规模调整。

## 22. 上线前检查清单

上线前必须确认：

- 数据库已备份。
- 环境变量已配置。
- HTTPS 正常。
- CORS 正确。
- Cookie secure 正常。
- MFA 可用。
- 登录限流可用。
- 客户端 refresh 可用。
- 授权吊销可用。
- 设备拉黑可用。
- 文件验签可用。
- 审计日志可用。
- 监控告警可用。

## 23. 运维禁止事项

禁止：

- 生产使用 debug 模式。
- 生产使用默认密码。
- 生产开放数据库公网端口。
- 生产开放 Redis 公网端口。
- 手动修改数据库状态不记录。
- 删除审计日志。
- 直接替换 MASTER_KEY。
- 未备份执行 migration。
- 绕过后台接口直接上传文件到对象存储后发布。
