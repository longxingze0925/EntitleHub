# 项目实施路线图

本文档用于规划后台系统从 0 到生产可用的实施路线。目标是让 AI 按阶段推进，避免一开始范围过大、模块边界混乱、安全能力后补。

## 1. 总体策略

采用四个原则：

```text
先闭环
再加固
后扩展
可回滚
```

第一版不追求功能全，而是追求核心链路正确：

```text
后台登录 -> 创建应用 -> 创建授权 -> 客户端激活 -> 设备绑定 -> session refresh -> 授权验证 -> 吊销生效
```

## 2. 里程碑总览

```text
M0 项目骨架
M1 后台身份和租户
M2 权限和团队
M3 应用和客户
M4 授权码和设备激活
M5 客户端 session 和 refresh 轮换
M6 设备黑名单和吊销联动
M7 版本包签名分发
M8 客户端 SDK
M9 安全测试和生产加固
M10 试运行和上线
```

## 3. M0 项目骨架

目标：

- 后端项目可运行。
- 前端项目可构建。
- 数据库可连接。
- 配置体系可用。

交付：

- 后端 `/health`。
- 配置加载。
- 错误处理。
- 日志。
- 数据库 migration。
- 前端登录页占位。

验收：

- 后端启动成功。
- 数据库连接失败时有明确错误。
- 前端 build 成功。

不要做：

- 不做业务功能。
- 不做复杂 UI。
- 不做权限系统。

## 4. M1 后台身份和租户

目标：

- 可以初始化第一个租户和 owner。
- owner 可以登录后台。
- 后台 session 可用。

交付：

- `tenants`
- `team_members`
- `admin_sessions`
- `admin_refresh_tokens`
- 登录接口。
- 登出接口。
- 当前用户接口。

验收：

- 不能使用默认固定密码。
- 密码 Argon2id。
- Cookie HttpOnly。
- 后台不保存 token 到 localStorage。
- refresh token 轮换。

## 5. M2 权限和团队

目标：

- 后台开始具备角色权限。
- 成员管理可用。

交付：

- roles。
- permissions。
- role_permissions。
- team_member_roles。
- 成员邀请。
- 角色修改。
- 禁用成员。

验收：

- viewer 不能创建应用。
- developer 不能看审计。
- 不能禁用最后一个 owner。
- 禁用成员撤销 session。

## 6. M3 应用和客户

目标：

- 租户可以创建应用。
- 租户可以管理客户。

交付：

- customers。
- applications。
- app_key。
- app_secret_hash。
- 应用签名密钥。
- 客户 CRUD。
- 应用 CRUD。

验收：

- app_secret 只展示一次。
- private key 加密存储。
- 同租户客户 email 唯一。
- 跨租户不可访问。

## 7. M4 授权码和设备激活

目标：

- 管理员可以创建授权码。
- 客户端可以用授权码激活。
- 激活后创建设备。

交付：

- licenses。
- devices。
- 创建授权。
- 吊销授权。
- 客户端激活接口。

验收：

- 授权码数据库只存 hash。
- 过期授权不能激活。
- 超过设备数不能激活。
- 激活成功创建设备。

## 8. M5 客户端 session 和 refresh 轮换

目标：

- 客户端有短 access token。
- refresh token 可轮换。
- session 可撤销。

交付：

- client_sessions。
- client_refresh_tokens。
- signing_keys。
- 公开 JWKS。
- client auth middleware。
- refresh 接口。
- heartbeat 接口。
- verify 接口。

验收：

- access token 15 分钟过期。
- refresh token 只存 hash。
- 旧 refresh token 复用撤销 session。
- 被撤销 session 不能访问。
- access token 签名带 `kid`，JWKS 可验签。

## 9. M6 设备黑名单和吊销联动

目标：

- 管理员可以控制设备。
- 吊销授权和拉黑设备立即影响客户端。

交付：

- device_blacklist。
- 拉黑设备。
- 解除拉黑。
- 解绑设备。
- 授权吊销联动撤销 session。

验收：

- 拉黑后 heartbeat 失败。
- 吊销授权后 verify 失败。
- 解绑后 refresh 失败。

## 10. M7 版本包签名分发

目标：

- 后台可上传版本包。
- 服务端签名。
- 客户端可查询和下载。

交付：

- release_files。
- releases。
- 文件上传。
- sha256。
- Ed25519/RSA 签名。
- signature_kid。
- 短时下载 token。
- latest release 接口。

验收：

- 文件 hash 正确。
- 签名可验证。
- SDK 能按 `signature_kid` 找到 JWKS 公钥。
- token 过期不能下载。
- 路径穿越失败。

## 11. M8 客户端 SDK

目标：

- 提供最小可用 SDK。

交付：

- activate。
- refresh。
- heartbeat。
- verify。
- check_update。
- download_update。
- hash 校验。
- signature 校验。
- 本地加密缓存。

验收：

- SDK 可完成完整激活和验证流程。
- access token 自动刷新。
- 并发 refresh 不复用旧 token。
- 文件篡改时验签失败。

## 12. M9 安全测试和生产加固

目标：

- 补齐安全边界。
- 准备试运行。

交付：

- MFA。
- CSRF。
- CORS 白名单。
- 安全响应头。
- Redis 限流。
- 审计脱敏。
- 请求签名。
- nonce 防重放。
- 部署配置检查。

验收：

- 安全测试清单通过。
- 跨租户测试通过。
- 权限测试通过。
- refresh 复用检测通过。
- 请求签名测试通过。

## 13. M10 试运行和上线

目标：

- 小范围真实使用。
- 验证稳定性和性能。

交付：

- Docker Compose 部署。
- HTTPS。
- 数据库备份。
- 日志。
- 监控。
- 告警。
- 上线检查清单。

验收：

- 连续运行 7 天无严重错误。
- 备份恢复演练成功。
- 关键接口 P95 符合目标。
- 出现错误能通过日志定位。

## 14. 每阶段 AI 执行方式

每个阶段开始前，先让 AI 输出：

```text
本阶段目标
涉及表
涉及接口
涉及文件
风险点
测试计划
```

每个阶段结束后，AI 必须输出：

```text
完成内容
修改文件
新增 migration
新增接口
安全检查
测试结果
剩余问题
下一阶段建议
```

## 15. 任务拆分粒度

正确粒度：

```text
实现后台登录
实现 refresh token 轮换
实现应用创建
实现授权码创建
实现客户端激活
实现设备拉黑
实现版本包签名
```

错误粒度：

```text
实现整个用户系统
实现完整授权平台
实现所有后台页面
一次性写完 SDK
```

AI 任务必须小，不然容易漏安全细节。

## 17. 风险清单

### 17.1 范围风险

风险：

```text
一开始做太多模块，导致所有模块都半成品。
```

控制：

```text
严格按里程碑推进。
每个里程碑必须可运行、可验证。
```

### 17.2 安全风险

风险：

```text
权限、租户隔离、session 撤销漏掉。
```

控制：

```text
每个模块必须补安全测试。
所有查询必须检查 tenant_id。
```

### 17.3 AI 开发风险

风险：

```text
AI 顺手重构、跳过测试、编造已完成。
```

控制：

```text
每次任务要求给出文件、命令、测试结果。
不要一次给 AI 太大任务。
```

### 17.4 客户端安全风险

风险：

```text
客户端被逆向，伪造请求。
```

控制：

```text
服务端做最终判断。
device keypair。
请求签名。
nonce 防重放。
session 可撤销。
```

## 18. 建议开发节奏

单人或 AI 主导开发：

```text
第 1 周：M0 + M1
第 2 周：M2 + M3
第 3 周：M4 + M5
第 4 周：M6 + M7
第 5 周：M8
第 6 周：M9 + M10
```

这个时间是理想节奏，实际应以质量为准。

## 19. 阶段完成定义

一个阶段完成必须满足：

- 代码能编译。
- migration 能执行。
- 核心接口可调用。
- 权限和租户隔离通过。
- 关键安全测试通过。
- 文档更新。
- 没有明显 TODO 阻塞主流程。

## 20. 最终 MVP 定义

真正的 MVP 不是后台页面完整，而是下面链路完整：

```text
管理员登录
创建应用
创建授权码
客户端激活
绑定设备
刷新 token
心跳验证
管理员吊销授权
客户端立即失效
上传版本包
客户端下载并验签
```

这条链路跑通，系统才算有核心价值。
